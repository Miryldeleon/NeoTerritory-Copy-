# Deployment Notes — NeoTerritory

Two surfaces are documented here:

1. **Local dev bootstrap** on macOS, Linux, and Windows (Tier-1 work in
   the previous round).
2. **AWS EC2 production deployment** (sketched in `docs/TODO/auth-and-deployment.md`
   §3; runbook lives there).

This file collects the deterministic pieces that don't fit naturally in the
TODO doc — the script-parity table, the CMake-generator decision matrix,
and the env-default contract every bootstrap path is expected to honour.

---

## 1. Bootstrap entry points

| Host shell        | Entry script                                  |
|-------------------|-----------------------------------------------|
| bash / zsh        | `./bootstrap.sh`                              |
| PowerShell        | `./bootstrap.ps1`                             |
| cmd.exe (legacy)  | `setup.cmd` (shim → `bootstrap.ps1`)          |
| Any with Node     | `npm run setup` (delegates to the right one)  |

All entry scripts accept the same component flags:

| bash flag           | PowerShell flag       | Meaning                  |
|---------------------|------------------------|--------------------------|
| `--no-frontend`     | `-NoFrontend`          | skip frontend deps       |
| `--no-backend`      | `-NoBackend`           | skip backend deps        |
| `--no-microservice` | `-NoMicroservice`      | skip CMake build         |
| `--no-infra`        | `-NoInfra`             | skip Docker/k8s          |
| `--only <name>`     | `-Only <name>`         | run a single component   |

`<name>` ∈ `frontend | backend | microservice | infra`.

## 2. CMake generator matrix

The microservice ships a vanilla `CMakeLists.txt`. Each host should pick
the generator that's already installed; the bootstrap scripts try the
first row and fall back through the others.

| Host            | First-choice generator                  | Fallback                       |
|-----------------|------------------------------------------|--------------------------------|
| Windows + VS    | `Visual Studio 17 2022` (`-A x64`)       | default (auto-detected by CMake) |
| Windows + Ninja | `Ninja Multi-Config`                     | `Unix Makefiles` (under MSYS)   |
| macOS           | `Ninja` if installed, else `Unix Makefiles` | Xcode (`-G Xcode`)            |
| Linux           | `Ninja` if installed, else `Unix Makefiles` | —                            |

To force a specific generator, pre-set `CMAKE_GENERATOR`:

```bash
export CMAKE_GENERATOR=Ninja
./bootstrap.sh --only microservice
```

```powershell
$env:CMAKE_GENERATOR = "Ninja Multi-Config"
.\bootstrap.ps1 -Only microservice
```

Build artefacts land in `Codebase/Microservice/build/` regardless of the
chosen generator.

## 3. `.env` default contract

Every bootstrap path is expected to seed (or preserve) the same backend
defaults so behaviour is identical across hosts:

| Key                    | Default                  | Notes                                          |
|------------------------|--------------------------|------------------------------------------------|
| `PORT`                 | `3001`                   |                                                |
| `CORS_ORIGIN`          | `http://localhost:3001`  |                                                |
| `JWT_SECRET`           | *(empty)*                | autogenerated on backend startup with warning   |
| `ANTHROPIC_API_KEY`    | *(empty)*                | optional; AI commentary gracefully degrades    |
| `NEOTERRITORY_BIN`     | *(empty)*                | optional; defaults to in-tree microservice build |
| `NEOTERRITORY_CATALOG` | *(empty)*                | optional; defaults to `Codebase/Microservice/pattern_catalog` |
| `SEED_TEST_USERS`      | `0`                      | `1` only in dev/test                            |

The bootstrap scripts copy `Codebase/Backend/.env.example` to
`Codebase/Backend/.env` only if `.env` does not already exist. They never
overwrite an existing file.

## 4. clean-browser.* parity

`clean-browser.sh` and `clean-browser.ps1` both launch a fresh-profile
Chromium with cache/cookies/history disabled, used by E2E tests and for
developer "fresh-state" reproduction. They accept the same args:

| arg              | meaning                                                 |
|------------------|---------------------------------------------------------|
| `<url>`          | URL to open (default `http://localhost:3001`)           |
| `--pw` / `-Playwright` | use the Playwright Chromium build instead of system Chrome |

Both scripts:
- create a `mktemp` / `New-TemporaryFile` profile dir
- pass identical `--disable-*` flags to the binary
- delete the profile dir when the browser exits

If you find behaviour drift, log it as a parity bug rather than fixing
silently — both should remain interchangeable.

## 5. AWS EC2 production deployment

See `docs/TODO/auth-and-deployment.md` §3 for the full runbook. Short
version: single-region t3.small (or graviton) running the Docker image
behind an ALB with HTTPS via ACM, SQLite on EBS or RDS Postgres for
multi-instance, Anthropic key in SSM Parameter Store, CloudWatch logs
via the awslogs driver, per-IP rate limit on `/auth/claim` and
`/auth/heartbeat`. The in-memory revocation list and seat queue must
move to Redis or DynamoDB before scaling past one instance.
